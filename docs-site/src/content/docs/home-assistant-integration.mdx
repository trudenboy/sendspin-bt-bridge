Available in **v2.65.0+**.

The bridge ships with two parallel transports for direct control from Home
Assistant dashboards and automations ‚Äî pick whichever fits your existing
setup, or use both together.

<Aside type="note">
  This integration **does not duplicate** Music Assistant's media surface.
  When sendspin-bt-bridge runs through Music Assistant, every Bluetooth
  speaker is already exposed in HA as `media_player.<name>` by the
  official Music Assistant integration.  Our integration only adds the
  things MA doesn't know about: BT signal strength, battery, sync
  health, idle behaviour, and BT-level commands like reconnect / wake /
  full reset.
</Aside>

## Two transports, one data model

|  | Path B ‚Äî MQTT discovery | Path A1 ‚Äî custom_component (HACS) |
|---|---|---|
| Lives in HA as | Auto-discovered via MQTT broker | HACS custom integration |
| Setup on HAOS | One-click (Mosquitto auto-detect) | One-click (Supervisor pair) |
| Setup off HAOS | Self-host MQTT (Mosquitto in Docker) | Generate token in bridge UI |
| Transport | MQTT pub/sub, JSON state per device | REST + SSE (`/api/ha/state`, `/api/status/events`) |
| Latency | Lowest | Low (SSE is push) |
| Source of truth | Same `services/ha_state_projector` projection | Same projection |

Both transports expose **the same set of entities**.  You'll see only
one device card per speaker either way ‚Äî entities declare
`connections=[("bluetooth", mac)]` so HA's device registry merges them
with MA's existing `media_player.<name>` card.

Plus a single bridge-level device card with `version`,
`ma_connected` binary sensor, `startup_phase` sensor, `update`
entity, `restart` and `scan` buttons.

## Path B ‚Äî MQTT discovery

### Setup on Home Assistant OS

1. Install the official **Mosquitto broker** add-on if you don't have
   one (Settings ‚Üí Add-ons ‚Üí Add-on Store ‚Üí Mosquitto broker).
2. Open the bridge web UI ‚Üí **Home Assistant** panel.
3. Click **Auto-detect MQTT add-on** ‚Äî the broker host / port /
   credentials are filled in automatically via the Supervisor API.
4. Toggle **Enabled** ‚Üí **mode: mqtt** ‚Üí Save.
5. Within seconds your speakers appear in HA under
   *Settings ‚Üí Devices ‚Üí MQTT*, **merged into the same device card MA
   already created**.

### Setup with a self-hosted broker

1. Run Mosquitto in Docker:

   ```yaml
   services:
     mqtt:
       image: eclipse-mosquitto:2
       ports:
         - "1883:1883"
   ```

2. In the bridge web UI ‚Üí **Home Assistant** panel, set the broker
   host:port and credentials manually, switch **mode** to `mqtt`, and
   **Enable**.

3. In Home Assistant, configure the MQTT integration to point at the
   same broker (Settings ‚Üí Devices ‚Üí Add Integration ‚Üí MQTT).

### MQTT topic shape

```
homeassistant/<component>/sendspin_<player_id>/<object_id>/config   (discovery, retained)
sendspin/<player_id>/state                                          (consolidated JSON, retained)
sendspin/<player_id>/availability                                   (online/offline, LWT)
sendspin/<player_id>/cmd/<command>                                  (subscribed)
sendspin/bridge/state                                               (bridge-level state)
sendspin/bridge/availability                                        (LWT)
sendspin/bridge/cmd/<command>                                       (subscribed)
```

## Path A1 ‚Äî custom_component (HACS)

### Setup

1. In HACS, add the bridge repository as a custom repository:
   `https://github.com/trudenboy/sendspin-bt-bridge` (category:
   *Integration*).
2. Install **Sendspin BT Bridge**.
3. Restart Home Assistant.
4. HA auto-discovers the bridge via mDNS and offers
   **"Sendspin BT Bridge discovered"** in *Settings ‚Üí Devices*.
5. **On HAOS** ‚Äî click *Configure*; the integration mints a token via
   the Supervisor pair endpoint with no further input.
   **Off HAOS** ‚Äî open the bridge web UI, *Home Assistant* panel, click
   **Generate token**, copy it once, and paste into HA's confirmation
   dialog.

### What you can automate

```yaml
# Auto-disconnect a speaker that's been idle for an hour
automation:
  - alias: "Park ENEBY20 after 1h idle"
    trigger:
      - platform: state
        entity_id: binary_sensor.eneby20_audio_streaming
        to: "off"
        for:
          hours: 1
    action:
      - service: button.press
        target:
          entity_id: button.eneby20_standby

# Force a full BT reset when reconnect attempts pile up
  - alias: "Reset ENEBY20 on reconnect storm"
    trigger:
      - platform: numeric_state
        entity_id: sensor.eneby20_reconnect_attempt
        above: 5
    action:
      - service: button.press
        target:
          entity_id: button.eneby20_reset_reconnect
```

## Authentication

`POST /api/auth/tokens` mints PBKDF2-SHA256-hashed long-lived bearer
tokens used by the custom_component.  The plaintext is returned **once**
and never persisted ‚Äî copy it immediately when you generate one in the
bridge web UI.

The `Authorization: Bearer <token>` header is accepted on every
`/api/*` route alongside the existing session cookie.

For HAOS users the integration uses `POST /api/auth/ha-pair`, which
validates the Supervisor proxy chain (`X-Ingress-Path` header from a
`172.30.32.2` source) and mints a token labelled `ha-custom-component`
without operator input.

## Disabling the integration

* **Path B** ‚Äî flip the **Enabled** toggle off in the bridge web UI.
  The publisher tears down cleanly and HA marks the entities as
  unavailable.
* **Path A1** ‚Äî *Settings ‚Üí Devices ‚Üí Sendspin BT Bridge ‚Üí Delete*.
  The integration is removed; revoke the corresponding token in the
  bridge web UI to invalidate it permanently.

## What's NOT here

This integration deliberately does not expose `media_player`, volume,
mute, transport, queue state, now-playing metadata, shuffle, repeat,
or sync-group membership ‚Äî those belong to Music Assistant's
integration.  Duplicating them would create entity collisions and
confuse automations (which `media_player` to target?).
